# 给仓库做基建的简单版

> 说明：原微信链接无法直接读取，页面抓取超时且搜索结果显示需要环境验证。这里参考可读到的腾讯技术文章《从微信后端仓库发展史谈谈单仓和多仓》的核心思路，结合我实习期间接触到的前端业务仓、D2C 工具仓、Chrome 插件、Skills 和 Agent 项目，整理一个更适合实际落地的轻量版仓库基建方案。

## 1. 基建的核心目标

仓库基建不是为了把规则做得很复杂，也不是为了证明某种仓库模式一定正确。它的目标很朴素：

1. 新人能快速跑起来。
2. 改动能被自动验证。
3. 低级错误不要进入 CR/MR。
4. 文档和经验能沉淀下来。
5. AI 辅助开发时有稳定上下文和验收标准。

仓库不是越大越好，也不是越小越好。关键是边界清晰、构建独立、协作稳定、演进可控。

## 2. 先定义仓库边界

每个仓库最少要说清楚三件事：

1. **这个仓库解决什么问题**：是业务页面、公共组件、工具插件、Skill 包，还是实验原型。
2. **这个仓库的产物是什么**：H5 页面、Chrome 插件、CLI 工具、文档站、npm 包，还是一组 prompt/rules/skills。
3. **这个仓库不负责什么**：避免把业务代码、调试脚本、个人笔记、一次性实验和长期工具全部混在一起。

如果边界不清晰，后面的 CI、文档、依赖、发布和权限都会变得混乱。

## 3. 最小目录结构

一个简单仓库可以先保持这样的结构：

```text
repo/
├── README.md              # 仓库说明：做什么、怎么跑、怎么验证
├── package.json           # 统一脚本入口
├── src/                   # 源码
├── scripts/               # 自动化脚本，比如生成、检查、发布辅助
├── docs/                  # 技术方案、使用说明、排查记录
├── tests/                 # 单测、集成测试、样例用例
├── examples/              # 最小可运行示例
├── .github/ or .ci/       # CI 配置
├── .cursor/ or .codebuddy/ # AI rules、skills、commands
└── CHANGELOG.md           # 重要变更记录
```

对前端业务仓，`docs/` 里至少要放业务背景、开发流程、本地调试、常见问题和发布注意事项。

对 AI 工具仓，`docs/` 里还应该放输入输出示例、工作流图、评测记录、失败案例和误报案例。

## 4. 标准化脚本入口

仓库里最好不要让新人到处找命令，先把常用动作收敛到 `package.json`：

```json
{
  "scripts": {
    "dev": "启动本地开发",
    "build": "构建产物",
    "lint": "代码规范检查",
    "typecheck": "类型检查",
    "test": "运行测试",
    "format": "格式化代码",
    "check": "lint + typecheck + test",
    "docs": "生成或预览文档"
  }
}
```

最重要的是 `check`。它相当于本地质量门禁，提交 CR/MR 前先跑一遍，能减少很多低级问题。

## 5. 质量红线先做轻量版

不要一上来就搞很重的流水线。简单版可以先做几条最有价值的红线：

1. **格式红线**：统一 prettier/eslint，避免风格争论。
2. **类型红线**：TypeScript 项目必须能通过 typecheck。
3. **构建红线**：主分支必须能 build 成功。
4. **测试红线**：核心工具函数、关键流程至少有基本测试。
5. **安全红线**：禁止提交 token、cookie、私钥、内部账号、生产接口密钥。
6. **产物红线**：不把临时产物、截图缓存、`node_modules`、大体积报告直接提交进仓库。

这些红线可以先只对新增代码严格执行，历史代码慢慢治理。不要一刀切卡死业务。

## 6. CI 从最小闭环开始

最小 CI 闭环可以是：

```text
install
-> lint
-> typecheck
-> test
-> build
```

如果是工具类仓库，可以再加：

```text
运行 examples
-> 生成一份样例报告
-> 检查输出文件是否存在
```

比如 Chrome 插件、D2C 工具、还原度对比工具，最好保留一两个固定样例，保证“脚本还能跑、报告还能生成、核心链路没有断”。

## 7. 依赖治理

依赖不要随手装完就不管，至少做到：

1. 新增依赖前先判断是否必须，能用标准库或已有依赖就不要引入新包。
2. 锁定 lockfile，保证别人安装出来的版本一致。
3. 大版本升级要单独记录原因和影响范围。
4. 对工具仓尤其要记录关键依赖的作用，比如 Playwright、pixelmatch、pngjs、Figma API SDK。
5. 如果依赖只用于脚本或开发期，要放到 devDependencies。

基建的目标不是“依赖越少越好”，而是让依赖来源、用途和升级风险可见。

## 8. 文档最小集

一个能长期维护的仓库，至少要有四类文档：

1. **README**：这个仓库做什么、怎么安装、怎么启动、怎么验证。
2. **开发指南**：目录结构、关键模块、常用命令、本地调试方式。
3. **使用说明**：面向使用者，给出输入、输出、示例和常见问题。
4. **决策记录**：为什么这么设计，为什么不用其他方案，后续还能怎么改。

对 AI 工具链项目，额外建议补：

1. Prompt / Skill / Rule 的触发条件。
2. Agent 工作流的输入输出。
3. 失败案例和误报案例。
4. 评测指标和固定 benchmark。

## 9. AI 友好的仓库基建

现在很多开发会直接在 IDE 里让 AI 改代码，所以仓库也要为 AI 准备上下文：

1. 在 `.cursor/rules`、`.codebuddy/rules` 或 `AGENTS.md` 里写清楚项目约定。
2. 把常见任务写成 commands 或 skills，比如“跑还原度对比”“生成报告”“修复 lint”。
3. 把项目架构、模块职责、禁忌写进文档，而不是只存在个人记忆里。
4. 对长任务准备 `memory/session-state.md` 之类的状态文件，避免 Agent 压缩上下文后忘记当前进度。
5. 对生成类工具加固定样例和验收脚本，避免 AI 只生成“看起来对”的代码。

这部分和实习期间做的 skills、mcp、openclaw、还原度对比、D2C 视觉门禁可以连起来讲：AI 不是替代基建，而是更依赖基建。没有规则、文档和验证，AI 只会更快地产生不可控结果。

## 10. 灰度推进，不要一把梭

仓库基建最容易失败的方式是“一次性上全套规范”，结果把业务开发全卡住。更合理的做法：

1. 先选一个小仓库或小模块试点。
2. 只卡新增代码，不强行重构全部历史代码。
3. 先提示 warning，再逐步升级为 error。
4. 先让本地 `check` 跑通，再接入 CI。
5. 记录被 CI 卡住的原因，反过来优化规则。
6. 有回退方案，避免工具问题阻塞业务上线。

基建不是为了证明规则严格，而是为了提升研发效能。能让开发更快定位问题、更少返工、更放心合入，才是有效基建。

## 11. 可以怎么写进简历

可以压缩成一句：

> 参与仓库工程化基建梳理，围绕目录结构、标准脚本、CI 检查、依赖治理、文档规范、AI rules/skills 和质量门禁，设计轻量级仓库治理方案，目标是提升新人上手效率、降低低级错误、沉淀团队经验，并让 AI 辅助开发具备更稳定的上下文和验证机制。

面试展开时可以讲三个重点：

1. 仓库基建的核心不是工具堆叠，而是边界、一致性和可验证。
2. 质量红线要灰度推进，不能一刀切阻塞业务。
3. AI 时代的仓库更需要 README、rules、examples、tests 和固定验收脚本，因为这些就是 Agent 的项目上下文。
