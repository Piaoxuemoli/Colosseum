# UI 风格规范

Colosseum 的 UI 应该像一个“AI 竞技场控制台”：深色、紧凑、可观战、信息密度高，但不要做成普通后台面板。

## 视觉方向

- 主基调：深色竞技场、玻璃拟态面板、柔和发光边框。
- 关键词：spectator、arena、live telemetry、agent thoughts、match control。
- 避免：默认 shadcn 灰白后台、过度渐变、花哨但影响可读性的动效。

## 色彩建议

- 背景：接近黑色或深蓝黑，如 `#070A12`、`#0B1020`。
- 主强调色：电蓝 / 青色，用于在线、流式、当前行动。
- 次强调色：紫色 / 琥珀，用于 Agent 思考、风险、等待。
- 危险色：红色只用于错误、淘汰、失败，不做普通强调。
- 卡牌和日志文本必须保证对比度，不能为了氛围牺牲可读性。

## 布局原则

- 观战页面优先三块：主赛场、右侧实时面板、底部/侧边日志。
- 右侧面板固定宽度，承载 scoreboard、agent list、thinking chain、event log。
- 对局中最重要的信息始终可见：当前阶段、轮到谁、可见公开状态、倒计时/等待状态。
- 赛后页面突出排名、关键转折、筹码趋势和完整日志入口。

## 组件约定

- 通用组件可以放在 `components/`，游戏强相关 UI 放在 `games/<game>/ui/`。
- 思考链展示要支持 streaming delta，不要等完整文本结束才显示。
- 动效只表达状态变化：当前行动、思考中、事件入队、对局结束。
- Framer Motion 可用于入场、状态切换和日志高亮；避免长时间循环动画污染观战。

## 文案与可访问性

- UI 文案中文优先，代码标识英文。
- 按钮必须表达动作结果，如“开始比赛”“保存 Profile”“查看回放”。
- 错误文案要给恢复路径，例如“LLM API key 缺失，请回到 Profile 配置”。
- 图表和日志不要只靠颜色表达胜负或风险。

## 复用旧项目

可以参考 `old/` 的牌桌、思考气泡、排名、筹码图和日志体验，但不要复制旧架构。新实现应符合 Next.js App Router、服务端 Route Handler 和游戏自治原则。
